# CircuitGate 用户访谈脚本

**版本**: 2026-05-11  
**用途**: 20-30 分钟问题发现访谈，用于验证硬件团队在打样前审查、BOM 风险、制造输出和协作审查中的真实痛点。  
**原则**: 不诱导用户说需要 AI，不推销产品，不让用户评价假想功能，优先追问最近一次真实经历。

## 访谈目标

- 复盘最近一次打样前审查或返板经历，定位高成本错误来源。
- 识别现有审查流程中哪些问题可由规则、工具输出或报告提前发现。
- 验证用户愿意为什么结果付费，而不是验证用户是否喜欢某个功能描述。
- 发现误报、隐私、责任边界和制造商差异对产品采用的影响。

## 访谈对象筛选

1. 你在过去 12 个月内是否参与过 PCB 打样、Bring-up、BOM 整理或制造文件交付？
2. 你们主要使用 KiCad、Altium、EasyEDA，还是其他 EDA 工具？
3. 你的团队规模、硬件角色和项目类型是什么？
4. 过去 12 个月内是否遇到过返板、装配失败、BOM 缺料、接口接错、制造文件遗漏或审查延迟？
5. 是否愿意基于一个真实项目描述最近一次审查流程？可以匿名，不需要分享机密文件。

## 访谈结构

| 阶段 | 时间 | 目标 |
| --- | ---: | --- |
| 开场与背景 | 3 min | 确认角色、项目类型、工具链和保密边界 |
| 最近一次项目复盘 | 8 min | 挖真实问题，不讨论解决方案 |
| 审查流程与协作 | 6 min | 找出现有流程、责任人、检查项和交付物 |
| BOM 与制造输出 | 5 min | 验证 BOM、Gerber、装配文件风险 |
| 付费与采用门槛 | 5 min | 验证预算、价值阈值、隐私和误报 |
| 收尾 | 3 min | 请求样本、转介绍和后续验证 |

## pain points 问题

1. 请回忆最近一次最接近返板、已经返板或差点制造失败的项目，当时发生了什么？
2. 这个问题是在原理图、PCB、BOM、制造输出、装配还是 Bring-up 阶段发现的？
3. 如果这个问题提前 1 天发现，你们能省下什么成本或时间？
4. 这个问题当时为什么没有被现有流程发现？
5. 哪类错误最让你担心：电源、接口、封装、BOM、制造文件、测试点、EMC/ESD，还是版本协作？
6. 哪些检查你知道应该做，但现实里经常被跳过？
7. 你们最不能接受的误报是什么？什么类型的提示会让你直接忽略工具？

## workflow 问题

8. 从原理图完成到下单打样前，你们通常有哪些固定检查步骤？
9. 谁负责最终检查？硬件工程师本人、团队 reviewer、外部顾问，还是制造商工程师？
10. 你们会保存 checklist、review comment、DRC/ERC 输出或制造商反馈吗？
11. 你们使用 Git、网盘、EDA 内置版本、邮件，还是聊天工具协作硬件文件？
12. 每次 PCB 修改后，哪些检查必须重新跑？哪些检查经常被忘记？
13. 对 KiCad 项目，你们会运行 ERC/DRC/jobset 吗？输出结果如何被记录和处理？
14. 你们最希望报告里保留哪些证据，才能让 reviewer 信任结果？
15. 如果一个检查只能提示“需要人工确认”，你希望它出现在报告里吗？在什么条件下有价值？

## BOM 与制造问题

16. 过去一年有没有遇到过 MPN 缺失、DNP 混乱、封装不匹配、替代料缺失或供应风险？
17. 你们会在 BOM 中记录 manufacturer、MPN、LCSC/DigiKey/Mouser 编号、替代料和生命周期吗？
18. 下单前谁检查 Gerber、drill、position、BOM、assembly drawing、stackup 和制造商规则？
19. JLCPCB、PCBWay、Eurocircuits 或本地工厂的制造约束是否会影响设计检查？
20. 哪些制造输出问题最适合自动检查，哪些必须人工确认？

## willingness-to-pay 问题

21. 如果有人在下单前帮你做一次可复核审查，你愿意为哪种结果付费？
22. 对 2 层、4 层或更复杂板卡，你认为一次审查节省多少钱或多少天才值得购买？
23. 你更偏好一次性审查服务、CLI 工具、GitHub Action、Web 报告，还是团队订阅？
24. 如果基础工具免费，哪些能力值得付费：私有规则、CI、报告历史、BOM 数据、人工复核、AI 摘要？
25. 你愿意先用免费审查换反馈和公开匿名案例吗？不能公开的边界是什么？

## security 与责任边界问题

26. 你是否能上传 KiCad 项目、Gerber、BOM 或 PDF 原理图到第三方服务？
27. 如果不能上传，是否可以接受本地 CLI 或 GitHub Action 在自己的环境里运行？
28. 报告里哪些内容不能离开公司或项目仓库？
29. 你希望工具如何声明责任边界：辅助审查、制造保证、认证检查，还是风险提示？
30. 如果工具给出错误结论，你能接受的恢复方式是什么：waiver、人工复核、规则禁用、报告备注，还是退款？

## 追问模板

- 这件事发生在什么时候？当时谁参与？
- 当时有没有留下文件、截图、issue、聊天记录或制造商反馈？
- 你们最后是怎么修复的？花了多久？
- 如果重新做一次，你希望哪个检查在什么时候提醒你？
- 这个提醒需要包含什么证据，你才会相信？

## 记录字段

每次访谈记录到 `research/user-interviews.md`：

- 受访者编号、角色、团队类型、EDA 工具、项目类型。
- 最近一次痛点、发生阶段、成本影响、现有流程缺口。
- 可自动化检查候选、必须人工确认项、不可接受误报。
- 付费信号、采用门槛、隐私边界、可提供样本。
- 可追溯证据链接或匿名说明。

## 成功标准

- 完成 30 位目标用户访谈后，至少能回答：
  - 哪 3 个细分场景最可能付费。
  - 哪 20 条规则最值得先实现。
  - 哪些检查必须保持本地运行。
  - 哪些误报会破坏信任。
- 在真实访谈完成前，不能把假设标记为已验证事实。
